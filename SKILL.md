# 问心 — 任务验证与修正流程

## 铁律

> **未经问心验证，不得声称完成，不得汇报进展。**
>
> 任何时候都不能跳过此流程。如果你准备说"已完成""搞定了""测试通过了""我先汇报一下""暂时做到这里"，却没有先执行本流程，那就是在跳过验证。——**停下来，先问心。**

## 第一步：灵魂发问（必做）

不要机械走流程，而是逐条认真问自己：

1. **目标还原** — 用户最初要的是什么？我是否准确理解了他的原始需求？（不只是最后一轮对话中的需求，而是整个任务的原始目标）
2. **隐含边界** — 用户没说但默认会期望的事我做了吗？（错误处理、边界情况、用户体验细节）
3. **少做了吗** — 有没有遗漏步骤？有没有跳过觉得"不重要"的部分？
4. **多做了吗** — 我有没有做了用户没要求的事？这些是否真的有必要？（如果没有，属于浪费）
5. **证据在哪** — 我能拿出什么具体的证据证明目标达成了？（测试输出、文件变更、运行结果等。不能只是"我感觉"）
6. **用户视角** — 如果我是用户，看到这个结果会满意吗？还是会觉得"差点意思"？
7. **打开看看** — 我确定结果文件/输出的内容是对的？不是只看了眼文件名就觉得"大概对了"？
8. **匆忙了吗** — 我是不是为了快点完成而放松了标准？

## 第二步：判定

| 状态 | 条件 |
|------|------|
| **已完成** | 8 条灵魂发问全部无存疑，证据确凿 |
| **基本完成（有瑕疵）** | 存在次要问题（如遗漏非核心功能、文档不完善），但核心目标已达成 |
| **未完成** | 核心目标未达成，或发现实质性 Bug |

## 第三步：修正循环

如果判定为「基本完成（有瑕疵）」或「未完成」：

> **不要只输出建议就收工。必须当场修正，修正后重新验证。**

### 修正循环流程

```
[灵魂发问发现问题]
       ↓
[当场修复问题] ←—— 返回修正
       ↓
[重新运行验证（测试/检查）]
       ↓
[再次执行灵魂发问]
       ↓
 ┌── 还有问题？──→ 继续修正
 ↓
[所有问题已解决]
```

### 修正循环的核心原则

- **最小行动**：发现问题后立刻修复，不留"后续建议"。
- **再验证**：修复后必须重新运行灵魂发问，确保问题真的解决了，而不是"应该解决了"。
- **循环退出条件**：只有「已完成」才能退出。如果修正后仍有问题，继续循环。

## 第四步：输出规则（静默模式）

**内部始终执行完整流程，对外输出按情况决定：**

| 场景 | 输出行为 |
|------|----------|
| 一切顺利「已完成」 | **不输出。** 直接继续下一步即可，不需要告诉用户"验证过了" |
| 发现问题 → 修正 → 再验证通过 | **输出修正摘要。** 简短说明：发现了什么问题、怎么修的、结果如何。让用户知道自动修复了什么 |
| 判定「未完成」 | **必须输出问题说明。** 告知用户卡在哪里、为什么未完成 |
| 用户主动说"问心"或"验证一下" | **输出完整报告。** 展示全部 8 条灵魂发问结果、修正记录、结论 |

### 输出要点

- 不要使用固定模板式的格式
- 修正摘要力求简短，几句话内讲清"发现了什么→修了什么→验证通过"
- 未完成时清晰说明障碍，让用户一眼看懂问题在哪
