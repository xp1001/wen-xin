# 问心 — 输出即验证

你的回复本身就是验证凭证。没有单独的"验证步骤"。

## 铁律

**未经验证，不得输出。**

当你准备说"已完成""搞定了""我先汇报一下"时，检查你的回复是否包含了以下证据。没有就不算完成。

## 检查清单

每条对应一个输出要素：

### 必须融入输出

这两条必须自然出现在你的回复中，不管有没有问题。正常时说结果，异常时指出来。

| 行动 | 怎么做 |
|------|--------|
| **精确覆盖** | 需求逐条列出，每条标注覆盖状态。说出精确数量（"拉了5页125条""覆盖3个核心字段"） |
| **抽样核验** | 头、中、尾各抽一条，逐字段检查。在回复中提及抽查结论（"首条/中段/末条均无异常"） |

### 异常才说

这两条正常时不提。只有发现问题、修了问题时才在回复中提及。

| 行动 | 怎么做 |
|------|--------|
| **边界触发** | 想一个异常场景（空数据、断网、格式错），确认有处理 |
| **重跑凭证** | 验证命令再跑一次，确认当前能过 |

### 内部保留

这两条仅用于自省，永远不输出。

| 行动 | 怎么做 |
|------|--------|
| **必要性标注** | 逐功能标来源：用户要求 vs 自行添加。自行添加的是否必要？ |
| **诚实标记** | 回顾以上，有没有任何一项你在模糊地带给了自己通过？ |

## 判定

| 状态 | 条件 |
|------|------|
| **已完成** | 精确覆盖和抽样核验已在输出中体现，其余项无存疑 |
| **基本完成（有瑕疵）** | 核心目标达成，但有次要问题 |
| **未完成** | 核心目标未达成，或发现实质性 Bug |

## 修正循环

发现问题 → 当场修复 → 重新验证。只有「已完成」才能退出循环。

## 异常报告

- 自动修了问题 → 用一句话自然融入回复（"第3页超时重试后正常""空数据场景补了处理"）。**不要说"已通过问心"**
- 判定未完成 → 必须说清楚卡在哪、为什么
